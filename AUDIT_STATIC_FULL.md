## КРИТИЧНЫЕ (CRITICAL) — краши, парс-ошибки, полностью сломанные механики

### C1. [enemy_champion.gd, строки ~357-368] — Парс-ошибка: неверный отступ, файл не загрузится
**Проблема**: `_combo_generation += 1` имеет лишний отступ — воспринимается как вложенный блок внутри `_combo_active = false`. GDScript не распарсит файл.
**Почему это баг**: Файл Champion (регулярный враг, Floor 6) не загрузится вообще. Любая комната с чемпионом крашит игру.
**Как исправить**: Выровнять `_combo_generation += 1` на один уровень с `_combo_count = 0` и `_combo_active = false`.

### C2. [enemy_drowned_one.gd, строка ~243] — Парс-ошибка: разрыв строки внутри выражения
**Проблема**: `int(severed_limbs.get(DamageZone.Zone.LEFT_LEG, false)) + \t int(severed_limbs.get(DamageZone.Zone.RIGHT_LEG, false))` — разрыв строки с табом внутри арифметического выражения без скобок или `\`.
**Почему это баг**: Drowned One не загрузится. Любой этаж с этим врагом крашит.
**Как исправить**: Объединить в одну строку или обернуть всё выражение в скобки.

### C3. [enemy_shadow_stalker.gd, строки ~248-252] — Парс-ошибка: неверный отступ в блоке dissolve
**Проблема**: `_dissolve_timer -= delta` и последующие строки имеют лишний отступ — они вложены под `sprite.modulate.a = move_toward(...)`, который является однострочным выражением.
**Почему это баг**: Таймер dissolve никогда не декрементируется. Shadow Stalker остаётся живым при 30% прозрачности навсегда. Возможен парс-краш.
**Как исправить**: Выровнять строки 249-252 на уровень строки 248 (внутри `if _dissolving:`).

### C4. [boss_satan.gd, строки ~183-186] — Сатана мгновенно переходит в Фазу 3
**Проблема**: `_update_phase` считает `hp_pct = current_hp / (_phase_1_hp + _phase_2_hp + _phase_3_hp)` = `400 / 1500` = `0.267`. Это меньше порога 0.33, поэтому Фаза 3 активируется на первом кадре боя.
**Почему это баг**: Игрок пропускает Фазы 1 и 2. Босс стартует сразу со всеми атаками.
**Как исправить**: Либо `torso_hp = _phase_1_hp + _phase_2_hp + _phase_3_hp` (= 1500), либо пересчитать пороги фаз под 400 HP.

### C5. [boss_consort.gd, строки ~99-168] и [boss_curator.gd, строки ~93-163] — `_physics_process` без `super`
**Проблема**: Оба босса полностью переопределяют `_physics_process` без вызова `super._physics_process(delta)`. Теряется вся базовая логика AI: навигация, обработка состояний, детекция, кулдауны, регенерация.
**Почему это баг**: Базовое поведение base_enemy утеряно. Любые будущие изменения базового класса не наследуются.
**Как исправить**: Добавить `super._physics_process(delta)` в соответствующем месте, либо продублировать ВСЮ необходимую базовую логику.

### C6. [boss_sister.gd, строка ~504] — `_disable_enemy` не вызывает `super`
**Проблема**: Переопределение `_disable_enemy` чистит диалог и выборы, но никогда не вызывает `super._disable_enemy()`. Базовый класс не устанавливает `_disabled = true`, не удаляет из групп, не останавливает бой.
**Почему это баг**: Sister продолжает преследовать и атаковать игрока после «отключения».
**Как исправить**: Добавить `super._disable_enemy()` в конце переопределения.

### C7. [boss_accountant.gd, строка ~330] и [enemy_demon.gd, строка ~314] — `HazardZone.new()` вместо инстанциации сцены
**Проблема**: `load("res://scripts/combat/hazard_zone.gd").new()` создаёт голый скрипт без дочерних узлов (CollisionShape2D и т.д.). Аналогично Demon вызывает `HazardZone.new()`.
**Почему это баг**: Ловушки Accountant и демон при смерти создают hazard zone без коллизии — механика полностью сломана. Возможен краш, если HazardZone не class_name.
**Как исправить**: Заменить на `load("res://scenes/combat/hazard_zone.tscn").instantiate()` (убедиться, что .tscn существует).

### C8. [melee_hit.gd, строки ~56-64] — Обход дерева для поиска врага завершается после 1 шага
**Проблема**: В `_on_area_entered`, цикл while проверяет `if enemy == null: return` после `node = node.get_parent()`. На первой итерации, если родитель hurtbox не в группе "enemy", функция немедленно возвращается, не проверяя вышестоящие узлы.
**Почему это баг**: Если hurtbox Area2D вложен в промежуточный узел (а не прямо в enemy), melee-удар ВСЕГДА промахивается. Это ломает melee-боёвку для определённых структур сцены.
**Как исправить**: Убрать `if enemy == null: return` из цикла. Проверять после цикла:
```gdscript
while node != null:
    if node.is_in_group("enemy"):
        enemy = node as Node2D
        break
    node = node.get_parent()
if enemy == null:
    return
```

### C9. [game_scene.gd, строки ~33-34, ~42] — Null reference краш при запуске забега и получении урона
**Проблема**: `gore_system` инициализируется через `get_node_or_null("/root/GoreSystem")` — может быть null. Вызов `gore_system.clear_room_effects()` без null-guard крашит. Аналогично `ScreenEffects.update_vignette()` вызывается без проверки autoload.
**Почему это баг**: Каждый старт забега и каждое получение урона — потенциальный краш.
**Как исправить**: `if gore_system: gore_system.clear_room_effects()`. Для ScreenEffects: `var se = get_node_or_null("/root/ScreenEffects"); if se: se.update_vignette(...)`.

### C10. [sfx_player.gd, строки ~70-81] и [music_player.gd, строки ~53-58] — Вся аудиосистема играет тишину
**Проблема**: `play_sfx` устанавливает `player.stream = _silence_stream` и воспроизводит его. Ни один реальный аудиофайл не загружается и не воспроизводится. То же для музыки.
**Почему это баг**: Игрок не слышит НИКАКИХ звуков эффектов и музыки на протяжении всей игры.
**Как исправить**: Загружать и воспроизводить реальные аудиофайлы по имени SFX/музыкального трека.

### C11. [loot_spawner.gd, строки ~288-294] — `_find_floor_manager` всегда возвращает null, пикапы ничего не дают
**Проблема**: Функция ищет узел в группе `"floor_manager"`, но FloorManager НИКОГДА не добавляет себя в эту группу (`add_to_group` не вызывается).
**Почему это баг**: Все пикапы, созданные LootSpawner, удаляются без начисления игроку лута (fallback: `pickup.queue_free()`).
**Как исправить**: Добавить `add_to_group("floor_manager")` в `FloorManager._ready()`.

### C12. [basement_manager.gd, строка ~117] — `change_scene_to_file` вызван напрямую, не deferred
**Проблема**: `get_tree().change_scene_to_file("res://scenes/ui/game_over.tscn")` вызывается синхронно. Это немедленно освобождает текущее дерево сцены.
**Почему это баг**: Все отложенные вызовы, твины и таймеры текущей сцены получают use-after-free. Краш или неопределённое поведение при смерти в подвале.
**Как исправить**: `get_tree().call_deferred("change_scene_to_file", "res://scenes/ui/game_over.tscn")`.

## ВЫСОКИЙ ПРИОРИТЕТ (HIGH) — серьёзные баги геймплея

### H1. [object_pool.gd, строки ~62-73] — Force-reclaim пропускает `_active.erase`, утечка слота пула
**Проблема**: `_return()` делает ранний выход если `_pool.has(instance)`, при этом НЕ вызывая `_active.erase()`. Пул теряет учёт «активных» экземпляров — размер пула некорректен.
**Почему это баг**: Со временем пул перестаёт создавать новые экземпляры, думая что все слоты заняты. Визуальные эффекты и снаряды перестают спавниться.
**Как исправить**: В `_return()` всегда выполнять `_active.erase(instance)` до раннего выхода.

### H2. [thrown_weapon.gd, строки ~382-407] — Discharge создаёт непулированные снаряды
**Проблема**: `_discharge()` инстанцирует снаряды напрямую через `preload().instantiate()`, минуя пул. При возврате в пул `get_parent().has_method("return_instance")` → false → `queue_free()`.
**Почему это баг**: Обход системы пулинга. Фреймовые просадки при инстанциации. Лямбды с захватом `fake_weapon` могут крашнуть при освобождении.
**Как исправить**: Использовать пул WeaponManager-а для снарядов разряда.

### H3. [weapon_manager.gd, строки ~147-151] — Снаряд теряется при невалидном current_scene
**Проблема**: Если `get_tree().current_scene` невалиден, функция `return`-ит, но снаряд уже удалён из родителя пула. Он — сирота, не в дереве, не в пуле.
**Почему это баг**: Постоянная утечка слота пула снарядов при смене сцены.
**Как исправить**: Перед return вернуть снаряд в пул: `add_child(proj); _projectile_pool.return_instance(proj)`.

### H4. [hazard_zone.gd, строки ~75-79] — Slow остаётся навсегда при смерти тела в зоне
**Проблема**: Если тело `queue_free()` внутри hazard zone, `body_exited` может не сработать. Slow-эффект навсегда остаётся на теле.
**Почему это баг**: После перерождения / смены комнаты, slow может персистировать. Для тел, которые reparent-ятся а не frees-ятся, баг 100% воспроизводим.
**Как исправить**: Отслеживать `tree_exiting` тел и чистить slow.

### H5. [run_state.gd, строки ~195-196] — `from_dict` создаёт дублирующие EventBus-соединения, старый RunState утекает
**Проблема**: `RunState.new()` в `from_dict` вызывает `_init()`, который коннектит EventBus-сигналы. Старый RunState не получает `cleanup()` — его обработчики остаются подключены.
**Почему это баг**: Каждый EventBus-сигнал обрабатывается ДВУМЯ RunState — старым (утекшим) и новым. Двойной счёт достижений/счётчиков + утечка памяти (signal connection держит ссылку).
**Как исправить**: Вызывать `cleanup()` старого RunState перед созданием нового через `from_dict`.

### H6. [save_manager.gd, строка ~48] — `DirAccess.remove_absolute` с `user://` путём молча не работает
**Проблема**: `remove_absolute()` ожидает абсолютный OS-путь, не `user://`.
**Почему это баг**: Сохранения забегов никогда не удаляются. При перезапуске может загрузиться устаревшее состояние.
**Как исправить**: `DirAccess.open("user://").remove("hotel_run.json")` или `DirAccess.remove_absolute(ProjectSettings.globalize_path(RUN_SAVE_PATH))`.

### H7. [run_state.gd, строки ~156-192] — `_upgrade_stack_counts` не сериализуется
**Проблема**: `to_dict()` не сохраняет `_upgrade_stack_counts`. `from_dict()` не восстанавливает.
**Почему это баг**: После загрузки сохранения diminishing returns сбрасываются. Игрок может стакать один и тот же апгрейд на полную мощность.
**Как исправить**: Добавить `"_upgrade_stack_counts": _upgrade_stack_counts` в `to_dict()` и восстановить в `from_dict()`.

### H8. [game_manager.gd, строки ~233-266] — `_handling_death` не сбрасывается в `restart_run`/`go_to_title`
**Проблема**: Флаг `_handling_death` устанавливается в `true` при смерти и сбрасывается только в basement-обработчиках. `restart_run()` и `go_to_title()` не сбрасывают.
**Почему это баг**: При следующей смерти обработчик видит `_handling_death == true` и возвращается раньше времени. Игрок умирает без реакции игры.
**Как исправить**: Добавить `_handling_death = false` в `restart_run()` и `go_to_title()`.

### H9. [seed_manager.gd, строки ~42-60] — Кешированные RNG-объекты мутируются, ломая детерминизм
**Проблема**: `get_room_rng()` возвращает один и тот же кешированный RNG для комнаты. `get_room_enemy_config` продвигает его состояние. `get_room_loot_config` получает уже мутированный RNG.
**Почему это баг**: Детерминизм гарантируется только при строгом порядке вызовов. Смена порядка = другой результат. Спидраннеры не могут делиться сидами надёжно.
**Как исправить**: Создавать копию RNG или всегда наследовать от seed напрямую, не возвращать мутабельный кеш.

### H10. [game_manager.gd, строки ~386-391] — Инстанциация скрипта вместо сцены для toast
**Проблема**: `load("res://scripts/ui/unlock_toast.gd").new()` создаёт RefCounted/Node без `@onready`-переменных. Если скрипт extends Control и имеет `@onready`-ссылки — все null.
**Почему это баг**: Toast либо крашит, либо не отображается. `@onready`-переменные не инициализируются.
**Как исправить**: `load("res://scenes/ui/unlock_toast.tscn").instantiate()`.

### H11. [boss_accountant.gd, строки ~207-216] — Фаза 3: золотой барраж — мёртвый код
**Проблема**: `elif _phase >= 3` на строке 213 недостижимо — `if _phase >= 2` на строке 208 уже ловит Фазу 3.
**Почему это баг**: Сигнатурная атака Фазы 3 (золотой барраж) никогда не выполняется. Фаза 3 идентична Фазе 2 по золотым броскам.
**Как исправить**: Изменить `elif` на `if` или реструктурировать условие.

### H12. [boss_champion.gd, строка ~544] — Инвертированная проверка скалярного произведения щита
**Проблема**: `dot > 0.0` проверяет, что facing и knockback сонаправлены. Но knockback направлен ОТ атакующего, а facing — К игроку. `dot > 0` означает удар СЗАДИ, не спереди.
**Почему это баг**: Щит блокирует атаки сзади вместо передних. Фронтальные атаки проходят насквозь.
**Как исправить**: Изменить на `dot < 0.0` (или отрицать `incoming_dir`).

### H13. [boss_champion.gd, строки ~220-221] — Смерть волны отслеживается через `tree_exited`
**Проблема**: `enemy.tree_exited` срабатывает при ЛЮБОМ выходе из дерева (reparent, сцена-чейндж, очистка), не только при смерти.
**Почему это баг**: Босс может пропустить волны или начать descent раньше времени. `wave_enemies_remaining` может уйти в минус.
**Как исправить**: Подключиться к сигналу смерти врага (например, `enemy_disabled`).

### H14. [boss_curator.gd, строка ~503] — Клон вызывает `super._ready()`, получая полную коллизию врага
**Проблема**: `if is_clone: super._ready(); return` — клон получает навигацию, хертбоксы, группу "enemy".
**Почему это баг**: Клон засчитывается как враг для AoE и подсчёта врагов. Игрок может целиться в клона вместо реального Curator.
**Как исправить**: Для клонов пропустить `super._ready()` или вызывать только минимальную инициализацию.

### H15. [boss_madame.gd, строки ~596-632] — Утечка SceneTreeTimer-ов в фазе 2
**Проблема**: `_continue_shard` создаёт ~150 SceneTreeTimer-ов за 2.5с боя и добавляет в `_shard_timers`. Таймеры никогда не удаляются из массива.
**Почему это баг**: Утечка памяти в течение боя с Мадам.
**Как исправить**: Удалять отработавшие таймеры из массива или использовать `_process` вместо таймеров.

### H16. [boss_madame.gd, строка ~822] — `max_clones -= 1` срабатывает дважды за оба отрубленных руки
**Проблема**: `_apply_mutilation_effects` проверяет `if severed_limbs[LEFT_ARM] or severed_limbs[RIGHT_ARM]`. После потери обеих рук условие истинно для ОБЕИХ рук, `max_clones` уменьшается на 2.
**Почему это баг**: max_clones уходит глубоко в минус, все клоны уничтожаются преждевременно.
**Как исправить**: Отслеживать, было ли уменьшение уже применено (флаг `_clones_reduced_for_arms`).

### H17. [boss_sister.gd, строки ~329-346] — Off-by-one в damage cap на ровно 10% HP
**Проблема**: При `torso_hp == hp_threshold` (ровно 10%) cap не срабатывает (условие `> hp_threshold`). Полный урон проходит — Sister может быть убита, лишая игрока шанса spare.
**Почему это баг**: На ровно 10% HP один удар убивает Sister, обходя защитный механизм.
**Как исправить**: Изменить условие на `>= hp_threshold`.

### H18. [boss_sister.gd, строки ~356-383] — Listen path: жёстко заданные 13с против 14.4с диалога
**Проблема**: 4 записи × (3.0 + 0.6) = 14.4с диалога, но await ждёт только 13.0с.
**Почему это баг**: Ending C срабатывает до завершения последней реплики — обрезает финальную строку.
**Как исправить**: Рассчитывать время динамически из очереди диалогов или использовать callback по завершении.

### H19. [arena_room.gd, строки ~99-100] — `tree_exited` получает освобождённую ссылку на врага
**Проблема**: `enemy.tree_exited.connect(_on_enemy_tree_exited.bind(enemy))` — в момент вызова enemy уже freed. `enemy in active_enemies` — неопределённое поведение.
**Почему это баг**: Враги не удаляются из `active_enemies`, волна арены никогда не завершается.

### H20. [floor_XX_config.gd, строки boss-комнат] — Типы боссов не в `EnemySpawner.ENEMY_SCENES`
**Проблема**: `"madame"`, `"gourmand"`, `"accountant"`, `"attendant_prime"`, `"champion"`, `"curator"`, `"consort"`, `"sister"`, `"satan"` — все отсутствуют в `EnemySpawner.ENEMY_SCENES`. Они есть в `FloorManager.ENEMY_SCENES`, но если используется не тот спавнер — босс не появится.

**Как исправить**: Добавить боссов в `EnemySpawner.ENEMY_SCENES` или унифицировать спавн через `FloorManager`.

### H21. [floor_06_config.gd, строка ~204] — Конфликт `"champion"`: обычный vs босс

### H22. [enemy_attendant.gd, строки ~189-208] — Heal channel в chase навсегда замораживает Attendant
**Проблема**: `_begin_heal_channel` вызывается из `_state_chase`, но `_healing_channel_timer` декрементируется только в `_state_engage`. Таймер никогда не доходит до нуля.
**Почему это баг**: Attendant навсегда замирает с `_is_channeling_heal = true` и нулевой скоростью.

### H23. [enemy_demon.gd, строки ~365-368] — Двойной decay knockback

### H25. [base_enemy.gd, строка ~156] — `receive_damage` крашит на неизвестной зоне

### H28. [player_controller.gd, строки ~124-127] — Формула урона может выдать отрицательное/бесконечное значение
**Проблема**: `amount *= (1.0 - reduction) * taken_mult` — если `reduction >= 1.0` или `taken_mult < 0`, результат бессмысленный. Нет clamp.
